# 実装計画ガイド

## 目的

finger-tracker の機能追加・バグ修正・リファクタリングの実装計画を構造化して管理する。

### ガイドの役割

- 現状とゴールのギャップ分析
- タスクの優先度付け
- スコープ定義
- 作業計画の文書化

## ディレクトリ構成

```
plans/
├── GUIDE.md           # 本ガイド
├── TEMPLATE.md        # 計画テンプレート
├── {active-plans}.md  # アクティブな計画
└── resolved/          # 完了済み計画
```

ライフサイクル: `plans/` → 完了後 `plans/resolved/` に移動

## 実装計画の種別

| 種別 | 説明 | 例 |
|------|------|---|
| bugfix | バグ修正 | 深度値0でクラッシュする問題 |
| enhancement | 機能追加・改善 | フィルタアルゴリズム変更 |
| refactoring | リファクタリング | モジュール分割の見直し |

## ドキュメント構成

| セクション | 必須/推奨 |
|-----------|----------|
| 概要 | 必須 |
| 現状分析 | 必須 |
| ゴール | 必須 |
| タスクリスト | 必須 |
| スコープ境界 | 推奨 |
| リスク | 推奨 |

## 記法規約

### メタデータ

HTMLコメントで記載:

```html
<!--
種別: plans
対象: {テーマ}
作成日: YYYY-MM-DD
ステータス: active / resolved
-->
```

### タスクリスト

チェックボックスで管理:

```markdown
- [ ] タスク1
- [ ] タスク2
  - [ ] サブタスク2-1
```

### スコープ境界

「やること」と「やらないこと」を明示:

```markdown
## スコープ
### 含む
- {やること}

### 含まない
- {やらないこと}
```

## AIエージェント向けルール

1. 計画作成前に、関連する既存コード・設計書を調査する
2. タスクは具体的で検証可能な粒度にする
3. 完了した計画は `resolved/` に移動する

## 品質チェックリスト

- [ ] 全必須セクションが埋まっている
- [ ] タスクが具体的で検証可能
- [ ] スコープが明確
- [ ] 関連ドキュメントへの参照がある

## テンプレートの使用

`TEMPLATE.md` をコピーし、プレースホルダを埋める。ファイル名: `{kebab-case-title}.md`
